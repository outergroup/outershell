File Associations

How Outer Shell chooses apps for files

Outer Shell treats file opening as a daemon service: apps describe the kinds of files they support, and outershelld resolves a file path into matching openers.

Draft space: this page has working explanatory copy, but it is meant to be easy to rewrite while evaluating the design.

The Model

This is loosely similar to how macOS thinks about document types and app handlers. The difference is that Outer Shell apps do not declare their file associations in static app metadata or an XML property list. They register content types and openers through the Outer Shell API.

Outer Shell uses content type identifiers such as public.text, public.markdown, and public.image. A content type can name parent types it conforms to, so a Markdown file can be treated as both public.markdown and public.text.

Apps register themselves as openers for content types. An app that registers for public.text can be offered for files that are known to be text, including more specific text types that conform to it.

1. Define types Content types describe names, extensions, exact filenames, MIME hints, and parent types.
2. Register apps Apps register opener records for the content types they can handle.
3. Infer a file Given a path, outershelld infers the content type from file evidence.
4. Return openers The daemon returns apps whose registered types match the inferred type graph.

Registered Content Types

A content type definition gives Outer Shell enough information to recognize files without making every app repeat the same logic. Built-in definitions cover common types such as text, Markdown, JSON, images, PDF, and ZIP archives.

Custom apps can add more definitions at install time or runtime by calling the API. A type definition can include:

  • a stable identifier, such as public.text or dev.example.source
  • a display name for people
  • parent content types it conforms to
  • filename extensions, without leading dots
  • exact filename matches, such as Makefile or README
  • MIME type hints
outerctl content-type add \
  --content-type dev.example.source \
  --name "Example Source" \
  --conforms-to public.text \
  --extensions example,exs \
  --filenames Examplefile \
  --mime-types text/x-example

Inferring a File's Type

Clients can ask for openers by path instead of deciding the type themselves. When a path is provided, outershelld looks at exact filename, filename extension, file magic, shebang, and text sniffing. This keeps type inference centralized.

The result is not just one label. Outer Shell expands the inferred type through its parent types, so a specific match can still use a more general opener.

Registering Apps as Openers

An opener registration says: this backend can open files of this content type. It also provides the user-facing app name, the app's socket path, and a URL template used to build the final navigation URL.

In macOS terms, this plays a similar role to declaring document handlers in an app bundle. In Outer Shell, that relationship is registered programmatically with outershelld instead. 

outerctl opener add \
  --backend org.outershell.Plaintext \
  --content-type public.text \
  --socket-path "$XDG_RUNTIME_DIR/org.outershell.Plaintext" \
  --name Plaintext \
  --url-template "?file={file}"

The {file} placeholder is replaced with the percent-encoded file path. If the opener has a socket path, Outer Shell uses it to route the navigation to the app backend.

Looking Up Openers

A caller can query by file path or by explicit content type. Path queries are the normal case for file browsers: the caller passes the path, and outershelld returns matching apps with resolved URLs. 

outerctl opener list --content-type public.text --file /path/to/README

The Files app uses this model when it builds its "Open With" menu. It does not need to know whether a file is text, Markdown, or a shell script. It asks the daemon for openers and then navigates to the selected app URL.

Why Centralize This?

Centralizing file associations lets Outer Shell behave more like a local environment. Apps describe what they can open, file browsers ask one service for answers, and specialized type logic lives in one place.

The important design boundary: apps register capabilities, but outershelld owns the decision of what kind of file a path represents.
